# Vector Search Specification

## Purpose

Defines the embedding storage layout, brute-force cosine similarity scan, RRF
hybrid fusion, and the additive `mode` parameter on `mem_search`. The goal is
meaning-based retrieval alongside the existing FTS5/BM25 lexical path, with
zero new Go module dependencies and zero schema migration.

---

## Requirements

### Requirement: Embedding storage — little-endian float32 BLOB

Embedding vectors SHALL be stored in the existing `memories.embedding BLOB`
column as a contiguous sequence of IEEE 754 single-precision (float32) values in
**little-endian** byte order. No length prefix, no framing, no base64 — raw bytes
only.

The `memories.embedding_model` TEXT column SHALL record the `ModelName()` of the
provider that produced the vector. The `memories.embedding_created_at` TEXT column
SHALL record the RFC3339 UTC timestamp at which the embedding was written.

A row is considered **unembedded** when `embedding IS NULL`. Rows with `embedding
IS NOT NULL` but with a `embedding_model` that differs from the currently
configured model are treated as **stale** and are re-embeddable via the backfill
predicate (see embedding-backfill spec).

No schema migration is required. The three columns (`embedding`, `embedding_model`,
`embedding_created_at`) already exist in the local store schema (schema.go:260-262)
and in `domain.Record` (memory.go:73-75). This change only populates them.

> Headless testable: yes — codec round-trip test.

#### Scenario: Codec round-trip — encode then decode yields cosine 1.0 with itself

- GIVEN a float32 vector `v` of length 256 generated by a provider
- WHEN `v` is encoded to little-endian bytes and stored in `embedding BLOB`
- AND the blob is read back and decoded to a float32 slice `v2`
- THEN `v2` has length 256
- AND `cosine(v, v2)` equals `1.0` (within float32 rounding tolerance of 1e-6)

#### Scenario: Dim mismatch is an error at decode time

- GIVEN a stored `embedding BLOB` of length `N` bytes where `N mod 4 != 0`
- WHEN the decode function is called
- THEN it returns an error (not a panic)

### Requirement: Brute-force cosine similarity scan

A cosine similarity function over the local store SHALL be implemented as an
app-side scan — no SQL vector extension, no ANN index. The scan:

1. Reads all rows from `memories` WHERE `embedding IS NOT NULL` AND
   `deleted_at IS NULL`, optionally filtered by `project`.
2. Decodes each `embedding BLOB` to `[]float32`.
3. Computes cosine similarity between the query vector and each row vector.
4. Returns the top-K rows ordered by descending cosine similarity.

The scan is **project-scoped**: when a `project` filter is provided, only rows
for that project are scanned. This matches the FTS path's project scoping.

Rows where the cosine denominator is zero (zero-magnitude vector) SHALL be
assigned similarity 0.0 and NOT returned in results (avoid NaN propagation).

The cosine function is defined as:
`cosine(a, b) = dot(a,b) / (|a| * |b|)`

> Headless testable: yes — unit test with synthetic vectors; integration test
> with embedded store.

#### Scenario: Cosine scan returns the most similar row first

- GIVEN a store with 3 rows for project "test": vectors A, B, C
- AND a query vector Q that is identical to B
- WHEN a cosine scan is executed with project "test", limit 3
- THEN the first result is the row with vector B (cosine 1.0)
- AND B appears before A and C

#### Scenario: NULL-embedding rows are excluded from cosine results

- GIVEN a store with 2 rows: row-1 has a valid embedding, row-2 has `embedding IS NULL`
- WHEN a cosine scan is executed
- THEN only row-1 appears in the results

#### Scenario: Zero-magnitude vector assigned similarity 0.0

- GIVEN a row whose stored embedding is all-zeros
- WHEN cosine is computed between the query vector and the all-zeros vector
- THEN the similarity is `0.0` and the row is not included in top-K results

### Requirement: RRF hybrid fusion — k=60

When `mode=hybrid` is requested, the search path SHALL:

1. Run the existing FTS5/BM25 scan — producing an ordered list of up to
   `limit * 2` candidates (ranked-list A, rank 1-based from highest BM25).
2. Run the brute-force cosine scan — producing an ordered list of up to
   `limit * 2` candidates (ranked-list B, rank 1-based from highest cosine).
3. Fuse the two ranked lists using **Reciprocal Rank Fusion** with `k=60`:
   `RRF_score(d) = sum_over_lists( 1 / (k + rank(d, list)) )`
   where `rank(d, list)` is the 1-based rank of document `d` in that list,
   or the list is skipped if `d` is absent.
4. Return the top-`limit` documents by descending RRF score.

The merge key for deduplication is `sync_id`. When the same `sync_id` appears
in both lists its RRF score is the sum of both rank contributions.

**Tie-break**: when two documents have equal RRF scores, they SHALL be ordered
by `sync_id` lexicographically (ascending). This ensures deterministic output
for identical inputs.

The FTS list and cosine list are computed independently; neither modifies the
other. BM25 scores and cosine scores are never normalized or combined directly
— only ranks are used in the RRF formula.

> Headless testable: yes — unit test with synthetic rank lists; integration
> test with paraphrase fixture.

#### Scenario: Hybrid finds a paraphrase that FTS misses

- GIVEN a store containing a memory titled "DPAPI-sealed config key" with content
  about keeping secrets off disk (no keywords matching the query)
- AND the memory has a valid embedding computed from that title + content
- WHEN `mem_search` is called with `query="how we keep the writer secret off disk"` and `mode="hybrid"`
- THEN the "DPAPI-sealed config key" memory appears in the results
- WHEN the same query is run with `mode="fts"`
- THEN the "DPAPI-sealed config key" memory is ABSENT from the results

#### Scenario: RRF fusion — document in both lists scores higher than in one list

- GIVEN two rank lists:
  - list A (FTS): [doc-1, doc-2, doc-3]
  - list B (cosine): [doc-2, doc-4, doc-1]
- WHEN RRF with k=60 is applied
- THEN doc-2 (rank 2 in A, rank 1 in B) scores higher than doc-1 (rank 1 in A, rank 3 in B)

#### Scenario: RRF tie-break is deterministic by sync_id

- GIVEN two documents with exactly equal RRF scores
- WHEN the fusion function is called twice with the same inputs
- THEN the output order is identical both times
- AND the document with the lexicographically smaller `sync_id` appears first

### Requirement: SearchFilter.Mode — additive field, zero-value safe

A `Mode` string field SHALL be added to the existing `SearchFilter` struct in
`internal/localstore/search.go`. The field is optional and zero-value safe:

| `Mode` value | Behavior |
|---|---|
| `""` (zero value) | FTS5/BM25 only — identical to today's behavior |
| `"fts"` | FTS5/BM25 only (explicit; same as `""`) |
| `"semantic"` | Cosine scan only (no FTS) |
| `"hybrid"` | FTS + cosine → RRF fusion |

An unknown `Mode` value (any string not in the table above) SHALL be treated as
`"fts"` (safe default). This prevents future callers from accidentally enabling
undefined behavior.

The `mem_search` MCP tool SHALL expose `mode` as an optional string parameter
defaulting to `""`. The handler maps it to `SearchFilter.Mode` and passes it to
`SearchMemoriesFiltered`.

> Headless testable: yes.

#### Scenario: mode="" (zero value) is byte-identical to mode="fts"

- GIVEN a store with 5 memories for project "test"
- AND existing `mem_search` acceptance tests that do NOT pass `mode`
- WHEN those tests run against the updated binary
- THEN all tests pass unchanged (no result set or ordering changes)

#### Scenario: mode="fts" produces identical results to mode="" for any query

- GIVEN any query string and project
- WHEN `SearchMemoriesFiltered` is called once with `Mode=""` and once with `Mode="fts"`
- THEN the returned slices are equal (same elements, same order)

#### Scenario: mode="hybrid" with no embeddings present silently returns FTS results

- GIVEN a store with memories that all have `embedding IS NULL`
- WHEN `SearchMemoriesFiltered` is called with `Mode="hybrid"`
- THEN results are returned (no error)
- AND the result set equals the FTS result set for the same query

#### Scenario: mode="semantic" with no embeddings returns empty results, no error

- GIVEN a store with memories that all have `embedding IS NULL`
- WHEN `SearchMemoriesFiltered` is called with `Mode="semantic"`
- THEN an empty slice is returned (no error)

#### Scenario: unknown mode value falls back to FTS

- GIVEN `SearchFilter{Mode: "turbo-mode"}`
- WHEN `SearchMemoriesFiltered` is called
- THEN the result is identical to calling with `Mode="fts"` (no error, FTS path)
